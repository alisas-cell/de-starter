# De-starter 设计说明

日期：2026-07-20  
状态：对话设计已批准，待书面复核  
目标仓库：`de-starter`

## 1. 产品定义

`de-starter` 是一个符合 Agent Skills 标准的通用 Skill。它帮助用户把购买或克隆的 Starter、Boilerplate、Template 安全接管为自己的产品。

它不是独立 Agent。现有 Codex Agent 负责理解目标、与用户沟通和执行工作；Skill 提供固定工作流、风险规则、报告模板和配套扫描脚本。

核心价值是让 Starter 接管过程可重复、安全、可审计、可验证，而不是隐藏模板来源或删除合法授权信息。

## 2. 目标

- 扫描不同技术栈项目中的品牌、作者、仓库、域名、邮箱、营销内容、Demo 和样例资产残留。
- 区分可安全替换内容、用户决策内容、高风险业务标识和受保护法律信息。
- 在修改真实工作区前输出审计实验报告和拟修改 diff。
- 让用户逐类选择删除、保留或替换内容。
- 支持真实品牌资料与中性占位符两种接管方式。
- 在用户确认前保持真实工作区零修改。
- 在确认后执行修改、项目验证、二次残留扫描和恢复材料生成。
- 支持 Git 和非 Git 项目。
- 以可安装、可测试、可公开发布的 GitHub Skill 仓库交付。

## 3. 非目标

- 不以删除 LICENSE、版权声明或合法署名为目标。
- 不承诺自动理解所有项目和框架的业务语义。
- 不在没有迁移方案的情况下修改数据库、支付、认证或公开 API 标识。
- 不把 `$PRIVATE_SOURCE_NAME` 对应的 private acceptance Starter 品牌词、目录或技术栈硬编码为通用规则。
- 第一版不创建独立 Custom Agent、MCP 服务或多 Skill Plugin。
- 第一版不深度支持所有框架；它提供通用扫描核心、Node/Next.js 深度适配和其他技术栈的安全回退。

## 4. 术语

- **Agent**：理解用户目标、调用工具、作出判断并完成任务的执行者。
- **Skill**：Agent 按需加载的工作流、规则、参考资料和可选脚本。
- **来源身份**：Starter 作者或发行方的品牌名、域名、邮箱、仓库和视觉资产集合。
- **目标品牌**：用户希望项目最终使用的产品和组织身份。
- **残留**：来源身份、模板营销内容、样例内容或不再适用于目标产品的配置。
- **业务标识**：可能进入数据库、支付平台、API、事件或持久化数据的稳定标识。
- **真实工作区**：用户当前项目目录。
- **临时副本**：在真实工作区外创建、用于生成拟修改 diff 的受控副本。

## 5. 总体工作流

1. 只读检查项目类型、清单文件、Git 状态、构建工具和验证命令。
2. 扫描来源身份候选，要求用户确认哪些名称、域名、邮箱和仓库属于 Starter。
3. 要求用户选择真实品牌模式或中性占位符模式。
4. 生成审计实验报告，列出风险、证据、保护项和需要用户决定的类别。
5. 让用户分别决定 Demo、示例博客、虚构评价、测试数据和样例素材的处理方式。
6. 在项目外建立临时副本，并只在副本中应用拟议修改。
7. 输出统一 diff、二进制变更清单、保护项和验证计划。
8. 暂停并等待用户明确批准。拒绝、排除文件或更改品牌资料都会使 Skill 重新生成 diff。
9. 批准前重新校验源文件哈希；任何变化都会使旧 diff 失效。
10. 在真实工作区应用已批准修改。
11. 执行项目验证、二次残留扫描和占位符检查。
12. 输出最终报告、保留项、人工处理项和恢复方法。

## 6. 架构

### 6.1 工作流层

`SKILL.md` 定义：

- 触发条件和不适用场景。
- 来源身份确认流程。
- 品牌资料选择流程。
- 风险分类规则。
- 两个强制审批门：清理范围确认和拟修改 diff 确认。
- 脚本调用顺序。
- 失败时必须暂停的条件。
- 最终报告和验证要求。

### 6.2 工具层

第一版工具使用 Python 3 标准库实现，不引入第三方运行时依赖。工具负责：

- 文件发现、忽略规则和文本编码识别。
- 品牌、域名、邮箱、仓库地址和文件路径匹配。
- 二进制资产清单、大小、哈希与文本引用分析。
- 项目清单和验证命令发现。
- Markdown 与 JSON 审计报告生成。
- 临时副本创建、拟议修改和统一 diff 生成。
- 源文件哈希验证、已批准补丁应用和反向补丁生成。
- 二次残留扫描和中性占位符检查。

脚本不负责决定含义不明确的内容是否应删除。它输出证据，Agent 根据 Skill 规则与用户共同决定。

### 6.3 规则层

`references/` 保存：

- 残留分类和风险判断规则。
- 法律与许可证保护规则。
- 品牌资料字段和上下文替换规则。
- 项目适配器约定。
- 审计、diff 和最终报告格式。

详细资料不与 `SKILL.md` 重复；`SKILL.md` 只说明何时加载相应参考文件。

## 7. 风险模型

| 等级 | 默认行为 | 典型内容 |
| --- | --- | --- |
| P0 保护项 | 不自动修改，只报告 | LICENSE、版权声明、第三方许可证、真实密钥、生产数据 |
| P1 高风险 | 单独确认并要求迁移与回滚方案 | 数据库标识、支付产品 ID、API 路径、环境变量名、认证配置、套餐键 |
| P2 用户决策 | 按类别说明影响并请求选择 | Demo、示例博客、评价、测试数据、样例媒体、演示依赖 |
| P3 安全替换 | 可进入拟修改 diff | 展示品牌、SEO、邮件署名、作者信息、原仓库链接、模板营销文案 |

风险由“位置、用途和依赖”共同决定，不能只由关键词决定。同一个 `starter` 字符串可以在首页文案中属于 P3，在 `starter_monthly` 套餐键中属于 P1，在 MIT LICENSE 作者信息中属于 P0。

## 8. 品牌资料

### 8.1 真实品牌模式

必填字段：

- 产品正式名称
- 产品短名称
- 主域名或正式网址
- 联系或支持邮箱
- GitHub 仓库地址
- 产品所有者或公司名称

可选字段：

- 文档名称
- 社交账号
- Logo 与社交分享图片
- 法律主体和公司地址
- Newsletter 发件人
- 各语言品牌写法

必填字段不完整时，Skill 暂停，不生成正式替换 diff。

### 8.2 中性占位符模式

占位符根据上下文选择可检索且语法有效的值：

- 展示名称：`Your Product`
- 包名和 slug：`your-product`
- 域名：`example.com`
- 邮箱：`support@example.com`
- 仓库：`https://github.com/your-org/your-product`
- 所有者：`Your Company`

Skill 同时生成机器可读的占位符清单，记录标识、占位类型、文件位置和后续替换状态。二次调用可以验证未替换占位符。

## 9. 来源身份发现

来源候选从以下信号综合推断：

- 包名、作者和仓库字段。
- README 标题、安装命令和克隆地址。
- 网站配置、Logo、页脚和邮件署名。
- SEO 标题、描述和社交分享资源。
- LICENSE 中的作者身份。
- 高频专有名称、域名和邮箱。
- 文件名、目录名和外部素材 URL。

第三方框架、支付商、云服务和依赖名称不会因为高频出现而自动成为来源身份。所有候选都必须由用户确认后才能作为清理匹配条件。

## 10. 审计实验报告

报告默认生成在项目外的临时运行目录，同时提供 Markdown 和 JSON：

- 运行 ID、时间、目标目录和源文件哈希摘要。
- 技术栈、包管理器、Git 状态和验证命令。
- 来源身份候选和用户确认结果。
- 目标品牌模式及字段完整性。
- 命中明细：编号、文件、行号、类别、风险、证据、推荐操作。
- Demo、内容、测试数据和二进制资产分类汇总。
- P0/P1 保护项及原因。
- 未确认项和无法自动判断项。
- 拟议验证计划。

报告不得输出密钥值。疑似密钥只显示文件位置、变量名和脱敏摘要。默认不读取 `.env`；`.env.example` 可以参与品牌和占位内容扫描。

## 11. Diff 与审批

临时副本只复制需要分析和拟修改的项目文件，并遵守忽略规则。生成结果包括：

- 修改、删除和重命名文件数量。
- 按类别整理的统一文本 diff。
- 二进制文件删除、替换和重命名清单。
- 保留的 P0/P1 项目。
- 验证命令及预期范围。
- 无法自动处理的人工操作。

用户可以完整批准、拒绝、排除文件或类别、修改品牌资料。任何范围变化都会生成新的运行 ID、源哈希和 diff。只有针对当前源哈希的明确完整批准才允许写入真实工作区。

## 12. Git 与非 Git 项目

Git 项目使用 Git 状态和 diff 辅助审计，但不得重置、覆盖或清理用户现有修改。

非 Git 项目使用文件哈希和临时副本生成预览。应用修改前：

- 保存计划修改文件的原始版本到项目外运行目录。
- 验证恢复材料能够读取。
- 生成反向补丁和恢复清单。
- 排除 `.env`、密钥和未修改文件。

如果无法创建恢复材料，Skill 必须停止。

## 13. 项目适配

### 13.1 通用核心

任何文本型项目都获得：

- 配置、源码、文档、路径、域名、邮箱和仓库扫描。
- 许可证保护。
- 外部素材引用和二进制清单。
- Git/非 Git diff。
- 用户排除规则与二次扫描。

### 13.2 Node/Next.js 深度适配

第一版识别：

- `package.json` 与包管理器锁文件。
- App/Pages Router、公共资源、国际化文件和 SEO 配置。
- React Email 或其他邮件模板。
- 测试、lint、类型检查和 build 脚本。
- 生成文件与源文件的关系。

### 13.3 安全回退

Python、Ruby、静态网站和未识别项目使用通用核心。适配器未识别时，Skill 不猜测框架专属删除操作，只输出通用发现和建议验证命令。

## 14. 失败处理

以下情况必须暂停：

- 扫描、报告或 diff 脚本失败。
- 真实文件与预览源哈希不一致。
- 来源身份或目标品牌未确认。
- 许可证义务不明确。
- P1 标识缺少迁移、兼容和回滚方案。
- 临时副本或恢复材料不可用。
- 报告可能泄露秘密。

暂停结果必须说明失败阶段、已完成的只读工作、未修改保证和恢复建议。不得把失败降级为未经确认的直接修改。

## 15. 验证策略

### 15.1 工具测试

- 品牌、域名、邮箱、仓库和路径匹配。
- 忽略目录、文本编码和二进制识别。
- `starter_monthly`、第三方名称和许可证误报保护。
- 密钥脱敏。
- Git 与非 Git diff。
- 文件变化导致旧批准失效。
- 反向补丁和恢复清单。
- 中性占位符检测。

### 15.2 测试夹具

- Node/Next.js 双语项目。
- Python 项目。
- 静态网站。
- 无 Git 项目。
- 多来源品牌项目。
- 含支付、数据库和认证业务标识的项目。

夹具全部使用合成内容，不包含购买的 Starter 源码。

### 15.3 Skill 行为评估

使用相同任务分别测试未加载 Skill 和加载 Skill 的 Agent 行为，验证 Skill 能够稳定做到：

- 修改前生成报告和 diff。
- 在两个审批门暂停。
- 保护 P0/P1 项。
- 不泄露密钥。
- 用户改变范围后重新生成 diff。

### 15.4 真实验收

当前 private acceptance Starter 是第一个私有真实验收项目。真实执行顺序为 dry-run、用户确认、应用、`pnpm lint`、`pnpm test`、必要时 `pnpm build`、二次残留扫描。公开仓库只保留脱敏报告摘要，不包含其源码或专有资产。

## 16. 仓库与发布

目标结构把开源项目资料与可安装 Skill 分开：

```text
de-starter/
├── skills/
│   └── de-starter/
│       ├── SKILL.md
│       ├── agents/
│       │   └── openai.yaml
│       ├── scripts/
│       └── references/
├── tests/
│   └── fixtures/
├── examples/
│   └── sanitized-report.md
├── docs/
│   └── superpowers/
├── README.md
├── CHANGELOG.md
└── LICENSE
```

`skills/de-starter/` 是符合 Agent Skills 规范的纯 Skill 文件夹，只包含运行所需的指令、脚本、参考资料和界面元数据。README、CHANGELOG、测试和设计文档保留在仓库根目录，不进入安装后的 Skill 上下文。

用户可以把 `skills/de-starter/` 复制或链接到 `$HOME/.agents/skills/de-starter`，然后通过 `$de-starter` 明确调用，也可以用匹配 Skill 描述的自然语言触发。

第一版采用 MIT License 发布。发布前必须确认仓库中没有购买 Starter 的代码、资产、秘密或无法再分发的材料。

## 17. 实施顺序

1. 使用官方 `init_skill.py` 初始化 `skills/de-starter/` 和界面元数据。
2. 建立合成夹具与失败测试。
3. 实现通用扫描、风险证据和脱敏报告。
4. 实现临时副本、diff、源哈希和恢复材料。
5. 实现 Node/Next.js 适配与验证命令发现。
6. 编写精简 `SKILL.md` 和按需参考资料。
7. 执行 Skill 行为评估并修正规则漏洞。
8. 对 `$PRIVATE_SOURCE_NAME` 对应的 private acceptance Starter 执行真实 dry-run。
9. 用户批准 diff 后清理 Starter 并完成项目验证。
10. 根据真实验收修正 Skill，生成脱敏示例和发布文档。

## 18. 验收标准

- 一句话请求能够触发 Skill 或通过 `$de-starter` 明确调用。
- 批准前真实工作区没有内容修改。
- 每次修改都有对应审计报告、当前源哈希和拟修改 diff。
- P0 项不会进入自动修改。
- P1 项没有迁移与回滚方案时不会进入自动修改。
- Demo、示例内容和样例资产由用户分类确认。
- 真实品牌资料缺失时，用户必须选择暂停或中性占位符。
- 报告不包含秘密值。
- Git 和非 Git 项目均可预览、应用和恢复。
- 修改后执行适用验证命令和二次残留扫描。
- GitHub 仓库不包含购买 Starter 的代码、资产或受限内容。
