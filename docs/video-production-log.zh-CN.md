# De-starter 自媒体过程素材记录

最终录制顺序、逐字口播和鼠标重点见[中文视频导演稿](video-director-script.zh-CN.md)。本日志保留历史 TDD、复审和发布轨迹，不把旧阶段测试总数改写为当前总数。

这份记录用于在开发过程中同步保存可公开讲述的证据。它不包含购买源码、私有绝对路径、密钥或未经脱敏的运行产物。

## 记录格式

每个关键节点记录五项：观众能学到什么、真实发生了什么、证据、截图候选、最终口播位置。

## 节点 1：为什么文件清理完成后仍要审计目录

- 观众收获：Git 不追踪空目录，但用户本机仍可能看到带有 Starter 名称的残留目录。
- 真实过程：第一轮精确 Apply 已正确完成 72 个路径变化，但独立复核发现一个已为空的源命名目录仍存在。
- 证据：文件发现数保持 523 → 227；目录残留作为独立指标为 1，不能混入文件统计。
- 截图候选：最终制作“文件痕迹”和“目录痕迹”双指标说明图，使用合成路径和脱敏数字。
- 口播位置：长视频“为什么不是扫描字符串后直接删除”之后；短视频问题转折处。

## 节点 2：为什么没有直接执行 `rmdir`

- 观众收获：真正通用的 Skill 需要把每一次删除变成可审批、可复现、可回滚的操作。
- 真实过程：采用独立 `cleanup_empty_dirs` 决策；成功清理将通过描述符相对的原子移动进入外部备份，而不是临时 Shell 删除。
- 证据：规格和实施计划均要求目录状态、预览结果、审批令牌、备份及恢复顺序绑定。
- 截图候选：事务流程图——“审计 → 预览证明为空 → 新 token → 移入备份 → 可恢复”。
- 口播位置：功能介绍中的“精确预览与回滚”段落。

## 节点 3：本轮 TDD 起点

- 观众收获：这不是针对一个项目补丁，而是在合成项目上先定义通用行为。
- 当前状态：规格和计划已获用户批准；运行代码尚未修改；Task 1 将先写失败测试。
- 已完成证据：旧能力以 `ImportError` 进入 RED；初版 126/126 后审查未放行；加固版 130/130 且独立复审零问题。
- 已完成截图：`docs/assets/video/04-empty-dir-red-green.png`，1600×900，视觉检查与隐私扫描通过。
- 口播位置：视频中简短说明“我怎么证明它不是只对这一个 Starter 有效”。

## 节点 4：测试发现了一个真实的计划问题

- 观众收获：TDD 不只是在找代码 Bug，也会提前暴露接口改动顺序不合理。
- 真实过程：目录审计聚焦测试通过后，全套测试有 6 项 CLI 生命周期失败；原因是审计 JSON 已新增目录字段，而读取端原计划到后续任务才更新。
- 决策：把最小 CLI 兼容前移到同一任务，让生产者和消费者一起升级；更完整的篡改测试仍留在后续 CLI 任务。
- 截图候选：脱敏后的“focused GREEN / full-suite 6 RED / compatibility GREEN”三段式证据。
- 口播位置：长视频“为什么它不是一段批量替换脚本”段落，用来说明全流程测试的价值。

## 节点 5：为什么测试全绿以后仍然需要独立代码审查

- 观众收获：126 项测试通过并不自动等于实现足够安全；审查需要寻找测试尚未覆盖的竞争条件和数据边界。
- 真实过程：独立审查发现初版目录扫描可能在并发新增子项或目录被替换成符号链接时接受不完整快照；还发现忽略/敏感目录递归哈希过深，以及特殊权限位无法被 CLI 读回。
- 处理方式：逐项验证审查结论，增加确定性 RED 测试，再改成描述符锚定和前后状态一致性检查；修复后必须重新生成审查包复核。
- 截图候选：不展示源码，使用“126 tests green ≠ review approved”的流程卡，加上 Critical/Important → RED test → re-review。
- 口播位置：安全能力介绍之后，用 15–25 秒解释“自动测试和独立审查各自解决什么问题”。

## 节点 6：中断后的证据应该怎么写

- 观众收获：开发过程可以恢复，但不能为了让故事更漂亮而补写自己没有亲眼看到的测试结果。
- 真实过程：Task 2 的测试文件先于生产代码出现，但代理会话在生成报告前中断；恢复代理能验证当前 44 项聚焦测试和 137 项全套测试，却无法恢复最初 RED 的终端输出。
- 记录原则：明确写成“测试先出现，但原始 RED 输出未保留”，不把推断冒充证据；代码仍需接受独立审查。
- 截图建议：这一段以口播和简短时间线为主，不展示内部会话或私有路径，避免喧宾夺主。
- 口播位置：可放在学习成长反思的 5% 段落，强调第一次做 Skill 时学到的证据意识。

## 节点 7：预览本身也必须防止路径被调包

- 观众收获：即使所有真实写入都放在 Apply 阶段，Preview 也会创建和清理外部副本，因此它同样需要防符号链接和竞态攻击。
- 真实过程：Task 3 初版能证明目录为空并绑定 token，但审查发现只固定末端目录不够；祖先目录若被换成符号链接，路径式删除可能指向错误位置。
- 修复方向：从预览根开始逐层用描述符和 `O_NOFOLLOW` 固定目录，只能相对已固定父目录删除末端空目录；同时把目录状态加入 `decision_hash`。
- 截图建议：最终安全流程图中把“Preview 只读/外部副本”旁边增加“仍需防路径调包”的小注释，无需展示攻击代码。
- 口播位置：介绍预览能力时补一句：“预览不改原项目，但预览工具自己也不能信任未经固定的路径。”

## 节点 8：安全修复必须重新走一次审查闭环

- 观众收获：审查发现问题后，不能只改代码并沿用旧结论；要补一个能稳定复现风险的失败测试，再生成包含全部修复的新审查包。
- 真实过程：新增了“预览祖先目录被替换成符号链接”的确定性测试；修复后改为从预览根逐层固定描述符、复核目录身份，并相对已固定父目录删除。目录状态也被加入 `decision_hash`。
- 当前证据：修复后聚焦测试 60/60、全套测试 144/144、`git diff --check` 通过；但新的独立复审仍未放行，说明“全绿”只是复审输入，不是最终结论。
- 截图建议：沿用安全流程卡，在“发现 Critical/Important”后补上“新 RED → 修复 GREEN → 新 review package → 复审”，避免展示攻击实现细节。
- 口播位置：节点 5 后面，用一句话交代“发现、复现、修复、复审”是一个完整闭环。

## 节点 9：第二次复审发现更窄的最终竞态窗口

- 观众收获：逐层固定路径仍不等于“可以安全删除”；最终检查和真正删除之间只要还是两个动作，就可能存在被调包的瞬间。
- 真实过程：独立审查用定向探针复现：在最后一次身份校验之后、目录删除之前替换末端或祖先，旧实现可能错误成功，最严重时会删除被换入的目录对象。
- 新设计：不再直接 `rmdir`。先把待清理名字以描述符相对的原子移动隔离到运行目录内的受控 staging，再验证被移动对象的身份、类型和空状态；若不匹配则安全恢复或保留隔离并失败关闭，未知对象绝不删除。
- 证据状态：第二轮修复已提交；新增 final-window、foreign-entry 和 stale-source 回归，聚焦测试 64/64、全套 148/148、diff check 通过。新的独立复审正在重跑原攻击探针，因此尚不写“最终通过”。真实 Starter 尚未进入新预览，更没有被修改。
- 截图建议：最终使用“检查后删除（存在窗口）→ 原子隔离 → 隔离后验证 → 成功保留/失败恢复”的简化流程图，不展示攻击命令。
- 口播位置：安全性章节的真实案例，用通俗比喻解释：“先验身份证再关门，中间仍可能换人；所以先把对象请进独立房间，再验一次。”

## 节点 10：为什么一个“去 Starter 痕迹”的 Skill 会做得这么复杂

- 观众问题：搜索、替换、删除几个文件看起来很简单，为什么需要这么多步骤？
- 核心回答：清理一个指定项目很简单；公开给陌生用户复用时，难点是证明“只删该删的、保护不该动的、审批后内容没有变化、失败还能恢复”。
- 三层能力：第一层发现痕迹；第二层区分残留与真实功能；第三层用精确预览、审批令牌、外部备份、验证和回滚安全执行。
- 产品表达：普通用户只需要经历“审计 → 看预览 → 批准 → 完成”，底层复杂度不应转嫁给用户。
- 诚实边界：当前安全强度高于普通一次性脚本；后续继续控制范围，不为展示技术而增加没有实际用户价值的机制。
- 口播位置：开场提出疑问后立即回答，或放在“它和一句提示词有什么区别”章节作为主线。

## 节点 11：市场对比章节的证据口径

- 用户要求：口播稿必须完整说明这次做的工作，相比公开可见的 Starter 去痕迹 Skill 或相邻清理工具强在哪里。
- 检索范围：2026-07-22 初步检索 GitHub、Skills.sh、Skillstore 和 Claude Code Playbooks 的公开页面，关键词覆盖 `starter`、`debrand`、`branding cleanup`、`boilerplate cleanup`。
- 初步结论：公开结果中能找到通用文件清理、dry-run/人工批准、代码重构和 Skill 模板，但暂未找到一个可以直接核实、同时覆盖“Starter 语义审计 + P0–P3 分级 + 精确审批 token + 外部预览 + 事务备份/回滚 + 目录残留 + 真实项目验收”的同类 Skill。
- 表达纪律：不能说“市场上绝对没有”；应说“在本次公开检索范围内暂未发现直接同类”，并把“对方公开写明的功能”“对方没有公开说明”“本项目已用测试/真实实验验证”分开。
- 计划写入的差异：不仅搜关键词；保护 LICENSE/真实功能；文件与目录双审计；行级语义编辑；缺少品牌信息时暂停或使用中性占位符；预览与源码分离；状态和 token 绑定；外部备份、失败回滚和恢复证据；篡改/符号链接/竞态失效关闭；P1 迁移与回滚说明；测试、独立复审和真实 Starter 前后对比；公开材料隐私脱敏。
- 口播位置：主视频独立章节“它和一句提示词、普通清理脚本有什么不同”，并在简介中放仓库链接和能力对比表。

## 节点 12：随机名字不等于原子“不覆盖”

- 观众收获：把隔离目录取成 UUID 能降低日常碰撞概率，但它不能作为安全证明；只要“检查目标不存在”和“执行移动”是两个动作，仍可能覆盖中途出现的对象。
- 真实过程：独立探针在 staging 目标检查后插入外来空目录，普通 POSIX `rename` 会替换它；另一个探针在最终链接检查后插入 symlink，旧状态哈希会跳过链接并生成 manifest。
- 修复方向：使用操作系统提供的 descriptor-relative、原子 no-clobber rename；平台或文件系统不支持时直接失败关闭，不回退普通 rename。状态哈希本身必须拒绝 symlink，不能只依赖更早的一次检查。
- 表达边界：这不是普通用户高频遇到的场景，而是公开写入工具需要明确的失效边界；视频不展示利用步骤，只展示为什么“随机名”和“先检查”不等于原子保证。
- 口播位置：可并入节点 9，用 10 秒补充“隔离房间也必须保证不会挤掉原来的住户”。

## 节点 13：失败现场也属于需要保护的数据

- 观众收获：安全不只发生在一次执行里；如果第一次失败时保留了未知对象，第二次重试也不能把这个失败现场当普通缓存删掉。
- 真实过程：最终独立审查确认原子 no-clobber 实现正确，但发现三个恢复/快照边界：Preview 根身份未贯穿绑定；同 run 重试可能清掉失败现场；文件状态哈希仍有 `lstat → 路径读取` 窗口。
- 修复方向：固定 Preview 根描述符与 inode；失败后把预览标记为 `recovery-required` 并禁止自动覆盖；文件哈希改为 `O_NOFOLLOW` 描述符读取并在前后复核身份。
- 审查结论：Critical 0、Important 3、Minor 0；67/67 与 151/151 虽通过，但 Task 3 仍未批准。
- 修复证据：三条边界先以 70 tests 中 3 条预期失败进入 RED；修复后聚焦 70/70、全套 154/154，均在 `ResourceWarning` 严格模式通过。新的完整独立复审正在进行，因此最终状态仍标为待批准。
- 口播位置：回滚能力章节，用一句话说明“可恢复不是有备份文件就够了，失败现场本身也不能被下一次运行吃掉”。

## 节点 14：审批令牌不仅绑定内容，也要绑定“这一次预览”

- 观众收获：内容哈希能发现文件变化，但 Preview 根在生成令牌的最后瞬间被替换时，还需要把这次预览目录本身的身份延续到 Apply 校验。
- 真实过程：恢复现场保护已经通过复审；剩余两项 Important 是 manifest 写入期间的根替换窗口，以及大文件分块哈希期间同 inode 原地改写。
- 修复方向：在写 manifest 后复核 Preview 根，并把根的 device/inode 加入 token-bound 契约，Apply 在任何备份或项目修改前核对；文件稳定性增加 size、mtime、ctime 前后比较。
- 表达建议：对普通观众可说“令牌既要对得上这份内容，也要对得上你刚刚看过的那份预览”；无需解释 inode 细节。
- 口播位置：精确审批 token 功能介绍中，用作“为什么批准后内容变化会自动失效”的例子。
- 最终证据：严格聚焦 74/74、全套 158/158；独立复审 Approved，Critical/Important/Minor 均为 0。Preview 根身份、文件稳定性、recovery-required、no-clobber 和旧 manifest 兼容均通过复核。

## 节点 15：内容完全一样，也可能不是原来那个目录

- 观众收获：内容哈希回答“里面是否一样”，inode 身份回答“是不是刚才批准的那个对象”；两者解决的问题不同。
- 真实过程：Task 4 写 Apply 复核测试时发现，旧 `cleanup_dir_states` 只有 mode、state hash 和 is_empty，无法识别 Preview 后被同内容、同权限的新目录替换。
- 决策：不降低已经批准的 inode-replacement 失效关闭要求；最小扩展 cleanup source identity，把 device/inode 纳入 token，Apply 在备份前用固定描述符核对。
- 兼容边界：没有目录清理的旧 manifest 继续兼容；历史 cleanup manifest 从未具备 Apply 能力，缺 identity 时明确拒绝，不能假装已验证。
- 口播位置：可放在 token 章节，用“箱子里的东西一样，不代表还是你刚才封条的那个箱子”解释。

## 节点 16：Apply 先完成体检，才有资格创建备份

- 观众收获：所谓“执行前复核”不是重新看一遍关键词，而是验证审批文件、源项目、外部预览和具体目录仍然是同一状态。
- Task 4 覆盖：严格 schema 和 token；路径重叠；mode/state/is_empty/device/inode；普通、ignored、secret 子项；文件/软链替换；Preview 根变化；末端和祖先目录改指；所有失败均不得留下 backup、restore 或 reverse artifact。
- 当前证据：RED 为 82 tests 中 6 failures + 2 errors；GREEN 为严格聚焦 83/83、全套 167/167，compileall 和 diff-check 通过；独立复审进行中。
- 关键边界：Task 4 只把完整体检放到正确位置，体检通过后仍会明确停在“Task 5 事务尚未实现”，没有提前执行删除。
- 口播位置：安全门介绍中，配合流程图说明“批准 token 不是永久通行证，执行前还要再验一次现场”。

## 节点 17：兼容旧格式不能变成降低新安全规则的后门

- 观众收获：向后兼容需要有边界；旧的无目录清理流程可以兼容，但新的目录清理不能通过删掉 identity 字段把自己伪装成旧格式。
- 真实过程：Task 4 独立审查发现 cleanup manifest 删除 `preview_root_identity` 后重新签 token 仍能被读取；还发现两处 raw FD 在 open 后的异常状态读取中泄漏。
- 修复要求：只允许 legacy no-cleanup 缺 identity；非空 cleanup 必须有完整 Preview 根身份。FD 必须在 open 成功的第一刻进入 `finally` 所有权，并用可计数回归测试验证，而不是只依赖 ResourceWarning。
- 审查结论：Critical 0、Important 2、Minor 0；普通测试 83/83 与 167/167 虽绿，Task 4 仍未批准。
- 口播位置：兼容性与安全边界章节，可用一句话：“兼容旧车票，不代表新路线可以不安检。”
- 最终证据：修复后严格聚焦 86/86、全套 170/170；独立复审 Approved，Critical/Important/Minor 均为 0。额外 after-stat 故障注入也确认两端 FD 为 EBADF。

## 节点 18：真正删除空目录，其实是把它搬进外部备份

- 观众收获：Task 5 的成功清理不调用 `rmdir`；目录从项目中消失，是因为它被原子移动到了外部 backup，因此“删除”和“可恢复”是同一个动作。
- 操作顺序：普通子项先进入 backup → 创建批准输出 → 再证明父目录为空 → 父目录进入 backup。回滚顺序反过来：先恢复父目录，再恢复子项。
- 当前 TDD：第一波 3 个成功/重建测试已 GREEN；第二波覆盖 child move 后插入外来项、最终 identity 变化、多目录中途失败、证据写入失败、rollback 目标被外来项占用。恢复证据字段先以 KeyError 进入 RED，再补 device/inode/source_was_empty/backup_empty 转绿。
- 安全语义：无法安全恢复时不覆盖现场，保留外部 backup，并明确报告 incomplete rollback，而不是为了“看起来恢复成功”删除外来内容。
- 口播位置：回滚功能核心演示；配一个“搬进保险柜，而不是扔进垃圾桶”的动画或流程图。

## 节点 19：原子动作完成后，要先记账再验货

- 观众收获：原子移动本身安全，不代表整个事务就安全；如果移动成功后先做可能失败的检查、晚一步才记录状态，异常时系统会误判“还没改过”。
- 真实过程：Task 5 审查探针复现目录已经进入 backup，但紧随其后的状态读取失败，旧 phase ledger 没有记录 mutation，因此没有触发 rollback；反向恢复和 `restore.json` 发布也有同类窗口。
- 修复规则：每个不可逆/原子 syscall 返回成功后，第一条逻辑必须登记 committed phase 或 ownership；后续验证失败才能根据真实阶段回滚或报告 incomplete。
- 审查结论：Critical 1、Important 1、Minor 1；常规 100/100 和 184/184 仍不足以批准事务。
- 口播位置：回滚原理章节，用“钱已经转出，必须先记流水，再查余额”解释事务账本。
- 最终证据：修复后严格聚焦 103/103、全套 187/187；原三个故障探针和额外五个事务场景全部符合预期，独立复审 Approved，Critical/Important/Minor 均为 0。

## 节点 20：先测试旧 Skill，再决定文案补什么

- 观众收获：Skill 指南不是凭想象堆规则；先让一个 fresh agent 面对压力场景，观察它真实漏掉什么，再针对失败补说明。

- 记录校正：实现者第一次进度消息手工误加为 7/9；逐项核对原始 9 项后确认 PASS 6、FAIL 3，仓库留证只采用 6/9。保留这条校正，避免把口头加总错误带进口播。
- 文案策略：只补这三类真实缺口，不重复已经做对的规则；forward 场景必须明确独立 decision、文件/目录双指标和事务恢复语义。
- 口播位置：介绍 Skill 为什么比一次性提示词稳定时，展示“baseline 6/9 → 指南修订 → forward 9/9”的证据卡。
- 单次开发轨迹：最小文案补强后曾得到 9/9；但独立审查指出每个版本仅一个 fresh-context 样本，不能排除偶然通过，因此该数字不作为最终效果证据，必须用 baseline/final 各 5 个独立样本重新验证。

## 节点 21：一次 9/9 不等于稳定 9/9

- 观众收获：Agent Skill 的输出有随机性；单次通过只能说明“这一次会”，不能证明“换个新对话仍会”。

## 节点 22：总审为什么还会找到分任务审查没找到的问题

- 观众收获：分任务测试通过不代表组合后没有旧路径；Task 5 给 cleanup directory 加了原子 phase ledger，但原有 ordinary child move 仍保留旧的“移动后再记账”和普通 rename。

- 口播位置：实验反思或安全总审章节，说明“局部正确”和“整条链正确”为什么是两件事。
- 最终修复证据：四条针对性回归先以 4/4 进入 RED；`3bbe94a` 将 ordinary Original 也改为 no-clobber 原子移动，并在 syscall 返回后第一时间记录 committed，同时把无 token 的 `preview.md` 纳入 artifact hash。严格 focused 108/108、全套 195/195、CLI 13/13 通过。

## 节点 25：真正公开后，Linux CI 推翻了一个 macOS 测试假设

- 首次公开状态：本地严格 195/195、CLI 13/13、干净公开克隆与安装副本均通过，但 GitHub Ubuntu 的 Python 3.11/3.13 运行在同一条 token 测试失败。

- 修复：比较两次 Preview 身份是否相同；身份相同就要求 token 相同，身份不同就要求 token 不同。同时单独比较 `decision_hash`，证明显式 `keep` 确实进入审批语义，而不是被运行路径变化掩盖。
